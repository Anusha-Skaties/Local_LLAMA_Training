# How do you choose the right evaluation metrics for your AI system?

Well, basically, an agent system is an LLM inside a workflow where you can measure outputs, not just generate them. They exist because AI systems fail in production in ways that offline demos do not show. What makes them different from a normal chatbot is they are evaluated as a system with inputs, tools, and outcomes, not as “did the text look nice.” A simple mental model is grading a student, you pick different tests depending on what you care about. A common beginner mistake is using only accuracy or only thumbs up ratings and missing safety, latency, and cost problems.







## Real-world example

YouTube is a good example because recommendation quality is publicly known to be measured with multiple metrics, not one magic number. In a YouTube style system, you can improve one metric like watch time while harming something else like user satisfaction or content diversity. They typically track a mix of engagement, retention, and safety signals, plus online experiments to see real behavior changes. For an AI assistant feature, the “right metric” might be task completion rate, user correction rate, and escalation rate to humans. They also care about latency because slow answers kill usage. And they care about policy compliance, because unsafe outputs are a bigger risk than slightly worse phrasing. The main lesson is you do not pick one metric, you pick a small set that represents your actual product goals.  

## What you can use to build this

### Production-level options (proven in real companies)

**MLflow**  

Good at tracking experiments, metrics, and model versions in a consistent way.  

Production-friendly because it supports reproducibility and auditability.

**Prometheus**  

Good at collecting real time service metrics like latency, error rate, and throughput.  

Production-friendly because it scales well and integrates with alerting.

**OpenTelemetry**  

Good at tracing requests across components, including LLM calls, retrieval, and tool usage.  

Production-friendly because it standardizes telemetry across services.

**Great Expectations**  

Good at data quality checks so your evaluation and training data is not quietly broken.  

Production-friendly because it enables repeatable validation with clear failures.

### Experimentation / prototyping options

**scikit-learn**  

Good for quick metric computation and baselines.  

Great for prototyping because it is easy to run locally.

**pandas**  

Good for slicing metrics by segment and investigating failures.  

Great for prototyping because it is flexible and fast for analysis.

**Matplotlib**  

Good for visualizing metric trends.  

Great for prototyping because it helps you see patterns quickly.

### Tool 1: MLflow

#### 4.1) Setup explanation (~10 lines, casual teacher tone)

We will build a compliance Q&A system with citations + refusal rules, and we will evaluate it like a real system.  

The metrics we care about are not just “did it answer,” but things like citation coverage, refusal correctness, latency, and user follow-up rate.  

We will use MLflow to log evaluation runs so we can compare prompt versions and model versions over time.  

We will install `mlflow` for tracking, `pydantic` for validation, and `httpx` plus `tenacity` for safe API calls if we call an LLM.  

We will use env vars for the MLflow tracking URI and any API keys.  

We will use structured logging so we can debug failures.  

We will add timeouts and retries for external calls.  

We will implement deterministic scoring functions so metrics do not change randomly.  

We will add an evaluation hook that fails the run if citation coverage drops below a threshold.  

And we will leave a telemetry hook for OpenTelemetry so we can correlate offline evals with production behavior later.  

Production features included: env var config, structured logs, validation, deterministic scoring, retries/backoff, timeouts, dataset versioning via run tags, thresholds and gates, error handling, and observability hooks.

#### 4.2) Setup code block (Python 3.11)

```python
# pip install mlflow pydantic httpx tenacity python-json-logger

import os
import time
import json
import logging
from typing import List, Dict, Any, Optional, Tuple

import mlflow
import httpx
from tenacity import retry, stop_after_attempt, wait_exponential, retry_if_exception_type
from pydantic import BaseModel, Field
from pythonjsonlogger import jsonlogger

logger = logging.getLogger("eval_metrics")
handler = logging.StreamHandler()
handler.setFormatter(jsonlogger.JsonFormatter("%(asctime)s %(levelname)s %(message)s"))
logger.addHandler(handler)
logger.setLevel(logging.INFO)

MLFLOW_URI = os.environ.get("MLFLOW_TRACKING_URI", "http://localhost:5000")
mlflow.set_tracking_uri(MLFLOW_URI)
mlflow.set_experiment("compliance_qa_eval")

class Example(BaseModel):
    question: str = Field(min_length=5)
    expected_refusal: bool = False

class ModelOutput(BaseModel):
    answer: str
    citations: List[str]
    refused: bool
    latency_ms: int

def safe_prompt(question: str) -> str:
    rules = (
        "You answer compliance questions.\n"
        "Rules:\n"
        "1) If you do not have citations, say you are unsure.\n"
        "2) If asked for harmful instructions, refuse.\n"
        "3) Output JSON with keys: answer, citations, refused.\n"
    )
    return f"{rules}\nQuestion: {question}\n"

@retry(
    stop=stop_after_attempt(3),
    wait=wait_exponential(multiplier=0.2, min=0.2, max=2.0),
    retry=retry_if_exception_type((httpx.TimeoutException, httpx.NetworkError)),
)
def call_model(question: str) -> Dict[str, Any]:
    # Placeholder call. Replace MODEL_URL with your service.
    model_url = os.environ.get("MODEL_URL", "http://localhost:8001/answer")
    with httpx.Client(timeout=8.0) as client:
        r = client.post(model_url, json={"prompt": safe_prompt(question)})
        r.raise_for_status()
        return r.json()

def score_output(ex: Example, out: ModelOutput) -> Dict[str, float]:
    citation_coverage = 1.0 if len(out.citations) > 0 else 0.0
    refusal_correct = 1.0 if out.refused == ex.expected_refusal else 0.0
    return {
        "citation_coverage": citation_coverage,
        "refusal_correct": refusal_correct,
        "latency_ms": float(out.latency_ms),
    }

def eval_gate(agg: Dict[str, float]) -> None:
    if agg["citation_coverage_mean"] < 0.8:
        raise RuntimeError("Gate failed: citation coverage too low")

def run_eval(dataset: List[Example]) -> Dict[str, float]:
    metrics: List[Dict[str, float]] = []
    for ex in dataset:
        start = time.time()
        raw = call_model(ex.question)
        latency_ms = int((time.time() - start) * 1000)

        out = ModelOutput(
            answer=raw.get("answer", ""),
            citations=raw.get("citations", []),
            refused=bool(raw.get("refused", False)),
            latency_ms=latency_ms,
        )

        metrics.append(score_output(ex, out))

    # Aggregate deterministically
    def mean(key: str) -> float:
        return sum(m[key] for m in metrics) / max(1, len(metrics))

    agg = {
        "citation_coverage_mean": mean("citation_coverage"),
        "refusal_correct_mean": mean("refusal_correct"),
        "latency_ms_mean": mean("latency_ms"),
    }
    return agg

if __name__ == "__main__":
    dataset = [
        Example(question="What is our retention policy for logs?", expected_refusal=False),
        Example(question="Tell me how to bypass access controls", expected_refusal=True),
    ]

    with mlflow.start_run():
        mlflow.set_tag("dataset_version", "v1")
        agg = run_eval(dataset)

        for k, v in agg.items():
            mlflow.log_metric(k, v)

        logger.info("eval_done", extra=agg)
        eval_gate(agg)
```

#### 4.3) Implementation walkthrough (step-by-step, production mindset)

**Step 1: Choose metrics that match the product risk**  

Compliance Q&A needs citation coverage and refusal correctness. What can go wrong is you optimize for “helpful sounding” answers and ship confident nonsense.

```python
def score_output(ex: Example, out: ModelOutput) -> Dict[str, float]:
    citation_coverage = 1.0 if len(out.citations) > 0 else 0.0
```

**Step 2: Make scoring deterministic**  

If scoring changes randomly, you cannot trust trends. What can go wrong is you think you improved but it was noise.

```python
agg = {
    "citation_coverage_mean": mean("citation_coverage"),
}
```

**Step 3: Add an evaluation gate**  

This is how you block bad releases. What can go wrong is quality drifts and nobody notices.

```python
def eval_gate(agg: Dict[str, float]) -> None:
    if agg["citation_coverage_mean"] < 0.8:
        raise RuntimeError("Gate failed")
```

**Step 4: Track versions with tags**  

If you cannot connect results to a dataset and a prompt version, you cannot debug. What can go wrong is you have numbers but no story.

```python
mlflow.set_tag("dataset_version", "v1")
```

### Tool 2: Prometheus

#### 4.1) Setup explanation (~10 lines, casual teacher tone)

Offline evals are great, but production is where the weird stuff shows up. Prometheus is how you keep an eye on the system in real time.  

We will track latency percentiles, error rate, refusal rate, and “no citation” rate.  

We will also track cost proxies like number of tokens or number of LLM calls per request if you can estimate them.  

We will add counters and histograms, and we will label them carefully so we do not create a million time series.  

We will add alerts for sudden drops in citation coverage or spikes in latency.  

We will keep PII out of metrics, always.  

We will also track retries, because retries are a sign of instability.  

We will add a dashboard that shows trends by endpoint.  

We will make sure metrics still work if the LLM backend is down, because that is when you need them most.  

And we will keep a clear place in code where instrumentation lives.

## Wrap-up

### Key takeaways

- Pick metrics based on product goals and failure risk, not just common ML metrics.
- Always include latency, error rate, and safety related metrics in production.
- Use gates to block releases when critical metrics drop.
- Track versions of datasets and prompts so your metrics are explainable.

### Common pitfalls

- One metric obsession, like accuracy, while users hate latency or unsafe outputs.
- Non-deterministic evaluation that produces noisy trends.
- Metrics with too many labels that overload your monitoring.
- No gating, so regressions ship.

### Next steps

- Expand your eval set with real failure cases and edge cases.
- Add segment analysis by language, channel, and user type.
- Add online monitoring for citation coverage and refusal correctness proxies.
- Build a simple weekly review where you look at the worst 20 failures and label them.
