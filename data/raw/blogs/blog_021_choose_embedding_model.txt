# How do you select the right embedding model for your use case?

Well, basically, an agent system is a setup where an LLM can retrieve context using embeddings, then answer with guardrails. They exist because most useful answers depend on your own docs, not the model’s memory. They’re different from a normal chatbot because they can look things up and cite sources instead of guessing. A simple mental model is “Google search, but the search is based on meaning,” and embeddings are how you store meaning. A common beginner mistake is choosing an embedding model only by benchmark score and ignoring language coverage, cost, and latency.







## Real-world example

Notion is a good single-company example because it’s publicly known as a knowledge product where search quality matters a lot. In a Notion style product, embeddings help with semantic search across pages, titles, and blocks. The “right embedding model” choice depends on whether you need multilingual search, how often content changes, and whether you need near real-time indexing. You’d also care about chunking strategy because Notion content is blocky, not just long PDFs. Latency matters because search should feel instant. And privacy matters because users might store sensitive notes, so you need a clear story for data handling. The embedding model is not just about quality, it’s about the whole indexing pipeline.

Good at running embedding workers and indexers reliably.

#### 4.2) Setup code block (Python 3.11)

```python
# pip install fastapi uvicorn psycopg[binary] pydantic pydantic-settings tenacity python-json-logger

# This is a minimal runnable skeleton showing how embedding selection impacts storage and routing.
# Real embeddings would come from an embedding model; we keep it deterministic here.

import time
import uuid
import logging
from typing import Optional, List, Dict, Any

import psycopg
from psycopg.rows import dict_row
from fastapi import FastAPI, Header, HTTPException
from pydantic import BaseModel, Field
from pydantic_settings import BaseSettings
from tenacity import retry, stop_after_attempt, wait_exponential
from pythonjsonlogger import jsonlogger

class Settings(BaseSettings):
    DATABASE_URL: str = Field(default="postgresql://postgres:postgres@localhost:5432/it")
    DB_STATEMENT_TIMEOUT_MS: int = Field(default=2000)

settings = Settings()

logger = logging.getLogger("it_router")
handler = logging.StreamHandler()
handler.setFormatter(jsonlogger.JsonFormatter("%(asctime)s %(levelname)s %(message)s %(request_id)s"))
logger.addHandler(handler)
logger.setLevel(logging.INFO)

app = FastAPI(title="Embedding Pipeline Skeleton")

class TicketRequest(BaseModel):
    subject: str = Field(min_length=3, max_length=200)
    body: str = Field(min_length=10, max_length=4000)

class TicketResponse(BaseModel):
    request_id: str
    routed_team: str
    similar_kb_ids: List[str]
    latency_ms: int

def db_connect():
    conn = psycopg.connect(settings.DATABASE_URL, row_factory=dict_row)
    with conn.cursor() as cur:
        cur.execute(f"SET statement_timeout = {settings.DB_STATEMENT_TIMEOUT_MS}")
    return conn

def fake_embed(text: str) -> List[float]:
    # Deterministic placeholder embedding
    s = sum(bytearray(text.encode("utf-8"))) % 1000
    return [s / 1000.0] * 8

@retry(stop=stop_after_attempt(3), wait=wait_exponential(multiplier=0.2, min=0.2, max=2.0))
def retrieve_similar_kb(subject: str, body: str) -> List[str]:
    # Placeholder keyword retrieval
    conn = db_connect()
    try:
        with conn.cursor() as cur:
            cur.execute(
                "SELECT kb_id FROM kb_articles WHERE content ILIKE '%' || %s || '%' LIMIT 5",
                (subject.split()[0],),
            )
            rows = cur.fetchall()
            return [r["kb_id"] for r in rows]
    finally:
        conn.close()

def eval_hook(team: str) -> None:
    if team not in {"it_helpdesk", "security", "network"}:
        raise RuntimeError("Unknown routing team")

@app.post("/route", response_model=TicketResponse)
def route(req: TicketRequest, x_request_id: Optional[str] = Header(default=None)):
    request_id = x_request_id or str(uuid.uuid4())
    start = time.time()

    logger.info("ticket_received", extra={"request_id": request_id})

    _ = fake_embed(req.subject + "\n" + req.body)  # where a real embedding call would occur
    kb_ids = retrieve_similar_kb(req.subject, req.body)

    routed_team = "security" if "phish" in req.body.lower() else "it_helpdesk"
    eval_hook(routed_team)

    latency_ms = int((time.time() - start) * 1000)
    logger.info("ticket_routed", extra={"request_id": request_id, "latency_ms": latency_ms, "team": routed_team})

    # Observability hook: track embedding latency, indexing lag, retrieval hit rate
    return TicketResponse(request_id=request_id, routed_team=routed_team, similar_kb_ids=kb_ids, latency_ms=latency_ms)
```

#### 4.3) Implementation walkthrough (step-by-step, production mindset)

**Step 1: Separate embedding generation from request path**  

If embedding calls are slow, your API becomes slow.

```python

```

**Step 2: Retrieval hit rate is a core metric**  

Bad embeddings show up as low hit rate or irrelevant hits.

```python

```

**Step 3: Evaluation hook**

#### 4.1) Setup explanation (~10 lines, casual teacher tone)

For embeddings, Kubernetes is mainly about workers. You run a queue consumer that reads new docs, generates embeddings, and writes them to the DB.  

You add resource limits because embedding workloads can blow up CPU and memory.  

You add autoscaling because indexing load comes in bursts.  

You add retries and dead-letter queues so you don’t lose documents.  

You add config separation with env vars.  

You add metrics for indexing lag.  

You add liveness checks so stuck workers restart.  

You add rollout strategy so updates don’t pause indexing for hours.  

You add network policies if you’re strict.  

And you keep logs structured so debugging is possible.

## Wrap-up

### Key takeaways

- Pick embeddings based on your retrieval needs: language, domain, latency, and update rate.
- The “best” embedding model is the one that works in your pipeline with your constraints.
- Track hit rate and relevance, not just offline benchmarks.
- Separate indexing from query serving.

### Common pitfalls

- Embedding everything as one huge chunk.
- No metadata, so filtering is weak.
- No measurement for retrieval quality.
- Calling embedding models inline on every request.

### Next steps

- Build a small labeled retrieval set for your domain.
- Add chunking rules and document versioning.
- Add dashboards for indexing lag and top failed docs.
- Add a reranker if relevance is still messy.
